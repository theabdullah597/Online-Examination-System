# Online Examination System - Project Overview

This document provides a comprehensive overview of the Online Examination System project located at `e:\Online Examination`.

## 1. Architecture & Tech Stack

The project follows a standard client-server architecture with a clear separation between the frontend application and the backend API.

### Backend (Node.js / Express / MongoDB)
- **Framework**: Express.js (v5)
- **Database**: MongoDB via Mongoose ODM.
- **Authentication**: JWT (JSON Web Tokens) with `jsonwebtoken` and `cookie-parser` for secure sessions.
- **Security**: 
  - `bcrypt` for password hashing.
  - `helmet` for setting secure HTTP headers.
  - `express-rate-limit` to prevent brute-force attacks.
  - `cors` configured to communicate securely with the frontend.
- **File Uploads**: `multer` (for handling multipart/form-data, e.g., possibly profile pictures or attachments).
- **Emails**: `resend` for sending emails (likely for registration, password resets, or exam notifications).

### Frontend (React / Vite)
- **Core**: React 19 built with Vite for fast HMR and optimized builds.
- **Routing**: `react-router-dom` for handling page navigation (SPA).
- **Styling/UI**: Custom CSS/Tailwind (implied by typical Vite setups, but definitely using standard React components), `lucide-react` for iconography.
- **Data Visualization**: `recharts` for displaying statistics, scores, or analytics on dashboards.
- **HTTP Client**: `axios` for making API requests to the backend.

## 2. Directory Structure

The project is divided into two main folders: `backend/` and `frontend/`.

### Backend
- `app.js`: Application setup, middleware integration, and route definitions.
- `server.js`: Entry point that binds the app to a port and starts the server.
- `models/`: Mongoose schemas defining the database structure.
- `controllers/`: Logic for handling incoming requests (e.g., `teacherController.js`).
- `routes/`: Express routers mapping URLs to controller functions (`authRoutes`, `adminRoutes`, `teacherRoutes`, `studentRoutes`).
- `middlewares/`: Custom middleware (e.g., auth verification, role checking).
- `config/`: Configuration files (e.g., database connection).
- `services/`: Business logic separated from controllers (e.g., email service).
- `utils/`: Helper functions.

### Frontend
- `src/pages/`: Contains all the main views of the application:
  - `Login.jsx`, `Dashboard.jsx`, `Classes.jsx`, `Teachers.jsx`, `Students.jsx`, `ManageExams.jsx`, `Questions.jsx`, `StudentExamInterface.jsx`, `Results.jsx`, `Submissions.jsx`, `SecurityLogs.jsx`.
- `src/components/`: Reusable UI components (`Header.jsx`, `Sidebar.jsx`, `Layout.jsx`, `ProtectedRoute.jsx`).

## 3. Database Entities (Models)

The database design revolves around several key entities:

1. **User**: Represents Super Admins, Teachers, and Students.
   - Fields: Full Name, Email, Password Hash, Role, Status, and student-specific fields like Roll Number and Class ID.
2. **Class**: Represents a group or grade (e.g., "Grade 10").
3. **Exam**: The core entity for assessments.
   - Fields: Title, Description, Class, Start/End Date, Duration, Total/Passing Marks, Status (draft, scheduled, active, completed), and flags for randomizing questions/options.
4. **Question**: Individual questions belonging to exams.
   - Types supported: MCQ, True/False, Fill in the Blank, Short Answer, Essay.
   - Includes fields for marks, difficulty, topic, and correct answer.
5. **Attempt**: Records a student's participation in an exam.
   - Tracks Start/End Time, Status (in_progress, submitted), and stores the student's answers along with marks obtained per question.
6. **AuditLog** & **SecurityLog**: For tracking system activity and security events (logins, unauthorized access attempts, etc.).

---

## 4. User Roles & Features

The system supports role-based access control (RBAC) with three primary roles:

### Super Admin
- Has full access to the system.
- Can manage teachers, students, and system configurations.
- Accesses security logs (`SecurityLogs.jsx`) and system-wide audits.

### Teacher
- Manages their assigned classes (`Classes.jsx`).
- Creates and configures exams (`ManageExams.jsx`).
- Adds and manages questions in the question bank (`Questions.jsx`).
- Reviews student submissions and grades manual answers like essays (`Submissions.jsx`, `Results.jsx`).

### Student
- Logs in to view their dashboard (`Dashboard.jsx`).
- Takes active exams via a dedicated, secure interface (`StudentExamInterface.jsx`).
- Views their past results.

---

## 5. Security Features
- **Passwords** are securely hashed using `bcrypt` before being stored in the database.
- **JWT** is used for maintaining user sessions, likely passed via secure HTTP-only cookies (indicated by `cookie-parser`).
- **Global Error Handling** is implemented in `app.js` to prevent app crashes and leakages of sensitive stack traces to the client.
- **Rate Limiting** is applied to the `/api` routes (max 100 requests per 15 minutes per IP) to prevent abuse and DDoS.
- **Helmet** adds standard HTTP security headers.

---

## 6. How to Run Locally

Both servers are currently running on your machine:
- **Backend**: Runs on Node.js using `npm run dev` (starts `nodemon server.js`). By default, it connects to MongoDB.
- **Frontend**: Runs using `npm run dev` (starts the Vite dev server). Typically accessible at `http://localhost:5173`.

The frontend communicates with the backend via Axios, and CORS is configured on the backend to accept requests from the frontend's URL.
